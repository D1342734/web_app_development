# 產品需求文件 (PRD) - 任務管理系統

## 1. 專案概述

### 背景與動機
在現代繁忙的生活與工作節奏中，學生與上班族每天都需要面對繁雜的課業、專案與日常瑣事。缺乏系統化的管理工具往往會導致任務遺漏、時間分配不均以及工作效率低下。本系統旨在提供一個輕量、直覺且高效的網頁版「任務管理系統」，協助使用者輕鬆記錄、規劃與追蹤所有待辦事項，進而提升生活與工作效率。

### 產品目標
*   **簡單易用**：提供直覺的操作介面，讓使用者能在數秒內新增、編輯或完成任務。
*   **即時狀態追蹤**：清晰展現任務的「待處理」與「已完成」狀態，讓使用者隨時掌握進度。
*   **彈性管理**：支援任務的篩選與搜尋，協助使用者在大量任務中快速找到目標，合理安排優先順序。

### 核心價值主張
「讓混亂的思緒變得井然有序。」本系統以最精簡的介面與核心功能，排除多餘的干擾，幫助使用者專注於「執行」與「完成」任務。

## 2. 使用者與需求

### 目標用戶
*   **學生**：需要記錄課業報告、考試日程、社團活動等事項。
*   **上班族**：需要追蹤每日工作待辦清單、會議時間與專案里程碑。
*   **個人管理者**：任何希望記錄生活瑣事、購物清單或習慣養成計畫的人。

### 使用者故事 (User Stories)
1.  **新增任務**：作為一名**使用者**，我希望能夠**快速新增一個任務並設定標題與描述**，以便我**不會遺忘需要做的事情**。
2.  **編輯任務**：作為一名**使用者**，我希望能夠**修改已建立任務的內容**，以便我**在計畫有變時能即時更新任務資訊**。
3.  **刪除任務**：作為一名**使用者**，我希望能夠**刪除不再需要的任務**，以便**保持任務列表的乾淨與整潔**。
4.  **標記狀態**：作為一名**使用者**，我希望能夠**一鍵標記任務為「已完成」或「未完成」**，以便我**能清楚了解目前的執行進度**。
5.  **顯示列表**：作為一名**使用者**，我希望能夠**在主頁面看到所有的任務列表**，以便我**對所有待辦事項一目了然**。
6.  **篩選任務**：作為一名**使用者**，我希望能夠**依照「全部」、「未完成」、「已完成」狀態篩選任務**，以便我**能專注於當前需要處理的工作**。
7.  **搜尋任務**：作為一名**使用者**，我希望能夠**透過關鍵字搜尋任務標題或內容**，以便我**在任務眾多時快速定位特定事項**。

## 3. 功能需求

以下為系統核心功能之詳細規格說明：

### 3.1. 新增任務 (Create Task)
*   **欄位說明**：
    *   `標題 (Title)`：必填，限制最多 100 字。
    *   `內容描述 (Description)`：選填，文字區域，限制最多 500 字。
    *   `建立時間 (Created At)`：系統自動帶入。
    *   `完成狀態 (Status)`：預設為「未完成」。
*   **操作流程**：使用者點擊「新增任務」按鈕，彈出表單或引導至新增頁面，填寫完畢後點擊儲存，系統寫入資料庫並重新載入列表。

### 3.2. 編輯任務內容 (Update Task)
*   **欄位說明**：可編輯 `標題` 與 `內容描述`。
*   **操作流程**：點擊特定任務的「編輯」按鈕，進入編輯表單，系統預先載入原任務資料。使用者修改後點擊「更新」，系統更新資料庫並返回列表。

### 3.3. 刪除任務 (Delete Task)
*   **操作流程**：點擊任務旁的「刪除」按鈕，系統跳出確認提示（例如：「確定要刪除此任務嗎？」）。使用者確認後，系統從資料庫移除該任務，並即時更新畫面。

### 3.4. 標記任務完成／未完成 (Toggle Task Status)
*   **操作流程**：在任務列表的每一項任務前提供核取方塊（Checkbox）或切換按鈕。點擊後，系統即時修改該任務的狀態（未完成 $\leftrightarrow$ 已完成），並更新資料庫。已完成的任務在視覺上應有明顯區隔（如標題加上刪除線、顏色變淡等）。

### 3.5. 顯示任務列表 (Read Tasks)
*   **介面設計**：主頁面以清單或卡片形式展示所有任務。
*   **資訊展示**：每項任務應顯示狀態、標題、簡短描述與建立時間。

### 3.6. 依完成狀態篩選任務 (Filter Tasks)
*   **操作流程**：介面提供切換標籤（Tab）或下拉選單，選項包含：
    *   **全部**：顯示所有任務。
    *   **未完成**：僅顯示狀態為未完成的任務。
    *   **已完成**：僅顯示狀態為已完成的任務。
*   **行為說明**：切換篩選條件時，頁面應即時或經由路由重新載入，顯示符合條件的任務。

### 3.7. 搜尋任務 (Search Tasks)
*   **操作流程**：介面頂部提供搜尋輸入框。
*   **行為說明**：使用者輸入關鍵字並按下 Enter 或點擊搜尋按鈕後，系統過濾標題或描述中包含該關鍵字的任務並進行顯示。可與篩選功能疊加使用。

為了快速驗證產品並提供核心價值，我們將功能區分為三個優先等級：

| 優先等級 | 功能項目 | 說明 |
| :--- | :--- | :--- |

## 5. 非功能需求

### 5.1. 技術限制與架構
*   **後端框架**：使用 **Python + Flask** 作為網頁伺服器與 API 開發工具。
*   **前端渲染**：使用 **Jinja2** 模板引擎進行 HTML 頁面渲染。
*   **資料庫**：使用 **SQLite** 進行輕量化資料儲存，不需額外安裝大型資料庫伺服器。
*   **樣式控制**：使用原生 CSS 進行視覺設計與排版，確保介面簡潔且載入迅速。

### 5.2. 效能與可用性
*   **響應式設計 (RWD)**：介面必須適應桌面端、平板端與行動端瀏覽器，確保學生與上班族在手機上也能順暢操作。
*   **頁面載入速度**：單頁任務數量在 100 筆以內時，首頁載入時間應小於 1 秒。

### 5.3. 安全與資料完整性
*   **輸入驗證**：所有前端表單與後端 API 均須進行輸入值驗證，防範 SQL Injection 與 XSS 攻擊。
*   **資料庫約束**：任務標題欄位設為 `NOT NULL`，確保資料完整。

## 6. 驗收條件 (Acceptance Criteria)

### AC 1: 新增任務
*   **情境**：使用者進入首頁，點擊新增按鈕。
*   **驗收標準**：
    1.  輸入有效標題與描述，點擊提交後，任務成功寫入 SQLite 資料庫。
    2.  頁面跳轉或更新後，新任務能顯示在列表中，且狀態預設為「未完成」。
    3.  若標題為空值點擊提交，系統應阻止提交並顯示錯誤提示「請輸入任務標題」。

### AC 2: 編輯任務
*   **情境**：使用者點擊某項任務的編輯按鈕。
*   **驗收標準**：
    1.  編輯頁面/表單能正確載入該任務的既有標題與內容。
    2.  修改內容並點擊保存後，資料庫成功更新，且列表顯示更新後的資訊。

### AC 3: 刪除任務
*   **情境**：使用者點擊某項任務的刪除按鈕。
*   **驗收標準**：
    1.  點擊後彈出二次確認視窗。
    2.  點擊確認後，資料庫中的該筆任務被物理刪除（或標記刪除），且列表不再顯示該任務。

### AC 4: 標記任務完成狀態
*   **情境**：使用者在列表中點擊任務狀態切換按鈕。
*   **驗收標準**：
    1.  點擊後，任務狀態即時在資料庫中由「未完成」改為「已完成」（或反之）。
    2.  已完成的任務在前端視覺上呈現出灰色字體與刪除線。

### AC 5: 篩選與搜尋
*   **情境**：使用者切換篩選標籤或在搜尋框中輸入關鍵字。
*   **驗收標準**：
    1.  選擇「已完成」標籤時，列表僅顯示已完成的任務。
    2.  輸入搜尋關鍵字「專題」，列表僅顯示標題或描述包含「專題」的任務。
    3.  若搜尋無相符結果，列表應顯示「查無相關任務」之提示訊息。
